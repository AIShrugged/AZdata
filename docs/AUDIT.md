# AZdata — Code Audit

**Date:** 2026-06-29 · Every finding in this report was independently re-checked and adversarially verified by a second agent before inclusion.

## Executive summary

AZdata is a working two-task LLM data system — natural-language-to-SQL over an e-invoice/taxpayer database (Task 1) and RAG-based Good/Service + product-group classification with HS-code assignment (Task 2) — fronted by a FastAPI service. The audit produced 36 confirmed findings: 2 high, 10 medium, 24 low, with no criticals. Honest verdict: this is a competent MVP/prototype that works on the happy path but is not production-hardened, and two of the findings are serious enough to block trust in the system as it stands.

The first high is a security hole at the exact place the design relies on for safety. The SQL guard that gates NL-to-SQL enforces a table/column allowlist and a row LIMIT but never inspects function calls, so a prompt-injected query using query_to_xml / database_to_xml / dblink can serialize the entire readable database in a single row — the allowlist and LIMIT are simply bypassed. The guard is also the sole control (no least-privilege DB role), and it has two further bypasses (schema qualifier ignored, column check not table-scoped). The second high invalidates the project's own scorecard: train/dev/test were split by row index without text dedup, so ~20% of dev/test items appear verbatim — with their gold labels — inside the train-based RAG retrieval corpus. That inflates both the headline Task-2 accuracy and the prompt-optimizer's reported gains, so the success metrics currently cannot be trusted as estimates of generalization.

Beyond those, the dominant pattern is silent failure: broad exception handlers and fallbacks that discard the model's real answer, cache failed classifications as successes, trust miscalibrated self-reported confidence, and count parse failures as 'Service'. The system tends to fail quietly-wrong rather than loudly. Reliability is also weaker than advertised — the 'local/cloud two-tier router' is actually cloud/cloud (a single OpenRouter dependency) and the embedding hot path has no timeout, so one upstream hiccup can both null every classification (then cache that null) and hang the whole synchronous FastAPI threadpool, including /health. The encouraging part, and the reason remediation is tractable, is that the codebase already contains the correct patterns it fails to apply consistently — a robust confidence clamp (classify._normalize_confidence), proper LLM timeouts/retries in nlsql.call_llm, a shared and unit-tested JSON parser (classify.extract_json), CLI provider validation, and a safe loopback default bind. Most fixes are small and amount to reusing what already exists and closing the two high-severity gaps.

## Top priorities

1. Close the SQL-guard function-call bypass (HIGH, security — src/nlsql.py guard_sql). The table/column allowlist and row LIMIT never inspect function calls, so a prompt-injected query using query_to_xml / database_to_xml / dblink exfiltrates the entire readable database in one row. Add an AST function allowlist (reject anything not on a vetted scalar/aggregate list; explicitly block *_to_xml, query_to_*, database_to_*, cursor_to_*, dblink*, pg_read_*, pg_ls*, lo_*, pg_sleep) AND enforce a least-privilege SELECT-only DB role so the guard is not the only control. Fix the schema-qualifier and column-scoping bypasses in the same pass.
2. Re-do the train/dev/test split and re-measure Task-2 before quoting any number (HIGH, ml_eval — scripts/make_splits.py). ~20% of dev/test items leak verbatim into the train-based RAG index with their gold labels, inflating both eval_task2 accuracy and the prompt-optimizer's dev gains. Split on unique normalized text (ideally grouped by source/invoice), assert zero train-vs-eval text overlap, then re-run eval_task2 and treat the prior reported metrics as invalid until reproduced.
3. Make the classification path resilient and stop masking failures (reliability cluster — src/router.py, src/rag.py, src/api.py). The advertised two-tier router is actually cloud/cloud on a single OpenRouter dependency; embed_texts has no timeout, so a stalled Ollama call hangs the whole synchronous FastAPI threadpool including /health; and total failures return label=None yet are cached as successes, persisting the outage after recovery. Add a genuinely independent local tier, pass timeout+retries to the embedding call, propagate ok/error state, and only cache real successes.
4. Lock down the public API surface (security/abuse — src/api.py). All endpoints are unauthenticated and unthrottled, CORS is '*', and clients pick the LLM provider/model, so an anonymous caller can burn the server's paid cloud keys and read all e-invoice data with no tenant scoping. Add auth + per-client rate limits, server-side allowlist the provider/model, restrict CORS to known origins, and return generic errors with a correlation id instead of leaking raw exception text and raw_sql. Keep the loopback default bind.
5. Verify and de-risk the HS-code deliverable (ml_eval + correctness — src/eqm.py). HS-code assignment has no ground truth and no evaluation at all, while the path has correctness bugs: a single wrong LLM heading permanently excludes the true code (embedding fallback fires only on an empty set), an out-of-list code is silently swapped for cands[0] but keeps the rejected code's confidence, and a non-numeric confidence string aborts an already-successful classification and substitutes a different code. Build a few-hundred-item gold set with top-1 and recall@k metrics, union heading-filtered with embedding candidates, and parse confidence defensively by reusing classify._normalize_confidence.

## Findings

36 confirmed findings: 0 critical, 2 high, 10 medium, 24 low. Each is tagged with its audit dimension.

### High

- **SQL guard does not restrict function calls — table/column allowlist and row LIMIT are bypassable via query_to_xml / database_to_xml / dblink** `security` — `/Users/frodobaggins/Dev/AZdata/src/nlsql.py` (guard_sql lines 214-255; _forbidden_classes lines 179-186) · guard_sql enforces single-statement, SELECT-root, a denylist of statement node classes (Insert/Update/Delete/...), a table-name allowlist via find_all(exp.Table), and a column allowlist. It never inspects or restricts function calls. PostgreSQL ships XML/table functions that execute an arbitrary SQL string passed as a literal argument: query_to_xml('SELECT ... FROM <any table the role can read>', true, false, ''), table_to_xml, cursor_to_xml, and database_to_xml(true,false,'') which serializes every readable table. Because the embedded query is a string literal, find_all(exp.Table) sees no Table node and the table allowlist is never applied; the function is parsed as exp.Anonymous, which is not in _forbidden_classes(). The outer 'SELECT database_to_xml(...) LIMIT 1000' returns a single row containing the entire readable database as XML, so the row LIMIT is bypassed too. The realistic trigger is prompt injection via the natural-language 'question', which steers the LLM to emit guard-passing-but-malicious SQL. The read-only session blocks writes but not these read functions; the only remaining control is PostgreSQL GRANTs. If the app DB role has broad SELECT (common for analytics), this reads arbitrary data outside the catalog. Other unrestricted functions enable DoS (pg_sleep, only partially capped by statement_timeout) and, with privileges, file reads (pg_read_file, pg_ls_dir, lo_get) or outbound connections (dblink). *Fix:* Add a function allowlist to the guard: walk the AST for exp.Func and exp.Anonymous and reject any function whose name is not on a vetted list (common aggregates/scalar funcs). Explicitly block *_to_xml, query_to_*, database_to_*, cursor_to_*, dblink*, pg_read_*, pg_ls*, lo_*, and pg_sleep. Independently enforce a least-privilege DB role (SELECT only on the catalog tables in a dedicated schema) so the guard is not the sole control.

- **Train/dev/test split is not deduplicated by text — ~20% of eval items leak verbatim into the RAG retrieval corpus with their gold labels** `ml_eval` — `/Users/frodobaggins/Dev/AZdata/scripts/make_splits.py` (main() lines 72-92 and _indices_for_split() lines 29-37 (split by row index, grouped only on ['label','group']); leaks via src/rag.py build_index() lines 50-61 (index built from train.csv) and retrieve() lines 75-80) · make_splits.py groups rows by (label, group) and shuffles ROW INDICES, never deduplicating identical item strings. labeled_items.csv is heavily duplicated (measured: 982 distinct texts occur more than once; 702 duplicate rows inside train alone). As a result identical item texts scatter across splits. Because the Task-2 classifier is RAG over the TRAIN index (rag.py build_index from train.csv, retrieve returns top-k train neighbors injected as labeled few-shot examples in build_rag_prompt), any eval item whose exact text is also in train is handed its own gold answer at inference time. I measured the overlap on the committed splits: 246/1261 unique dev texts (19.5%) and 256/1268 unique test texts (20.2%) appear verbatim in train. This inflates BOTH the eval_task2 headline accuracy and the optimize_prompt.py dev accuracy, so the reported Task-2 numbers are not a valid estimate of generalization. *Fix:* Split at the level of unique normalized text (and ideally by 'source'/invoice, which is already a column) using a grouped split so every row sharing a text lands in the same split; or deduplicate texts before splitting. After re-splitting, additionally drop any exact train/eval text collisions from the eval set (or from the index) and assert the measured train-vs-dev/test text overlap is 0 before trusting any accuracy number.

### Medium

- **No authentication or rate limiting; wildcard CORS; client-chosen LLM provider/model enables unauthenticated abuse of paid APIs and full data read** `security` — `/Users/frodobaggins/Dev/AZdata/src/api.py` (CORS middleware lines 46-52; /query lines 105-130; /classify lines 133-182; host env line 202) · All endpoints are unauthenticated and unthrottled, and CORS is allow_origins=['*'] (any web origin can call them from a victim's browser; credentials are off, so this is abuse rather than credential theft). /query forwards client-supplied req.provider and req.model straight into answer()->call_llm, and /classify forwards req.local_model/req.strong_model into router.classify_item; these can select cloud providers (openai/anthropic/openrouter) that spend the server's API keys. An anonymous caller can therefore force the most expensive cloud model on the server's account (financial DoS / open LLM proxy), run unbounded DB queries, and read all e-invoice/taxpayer data with no row-level or tenant scoping. The CLI restricts --provider to choices=sorted(DEFAULT_MODELS) (nlsql.py line 348) but the HTTP layer performs no equivalent validation. Default bind is 127.0.0.1 (good), but AZDATA_API_HOST is operator-overridable to 0.0.0.0, at which point this becomes high severity. *Fix:* Require an API key/auth on /query and /classify and add per-client rate limiting/quotas. Constrain provider to a server-side allowlist and ignore or allowlist the client-supplied model/local_model/strong_model (map to a fixed safe set) so callers cannot pick arbitrary paid models. Restrict CORS allow_origins to the known front-end origin(s). Keep the bind on loopback unless fronted by an authenticating reverse proxy.

- **Two-tier router is cloud/cloud: single OpenRouter dependency with no local or degraded fallback** `reliability` — `/Users/frodobaggins/Dev/AZdata/src/router.py` (module constants lines 10-13; classify_route lines 34-58; classify_item HS call lines 81-88) · PROVIDER is hardcoded to "openrouter" and both LOCAL_MODEL (qwen3.5-35b) and STRONG_MODEL (qwen3.5-122b) are OpenRouter-hosted. classify_route issues BOTH the 'local' tier and the escalated 'strong' tier through the same provider (provider=provider for both rag.classify_rag calls), and classify_item routes HS assignment through provider=route_kwargs.get('provider', PROVIDER)=openrouter. So the advertised 'local/cloud two-tier router' is actually cloud/cloud — every tier depends on OpenRouter. There is no real local tier and no degraded mode: if OpenRouter is unreachable or OPENROUTER_API_KEY is unset (nlsql.call_llm builds the client with api_key=os.environ.get('OPENROUTER_API_KEY'), which may be None), the local tier fails (ok=False) -> escalates -> strong tier fails -> classify_route returns the failed local result, so the entire Task 2 path returns null labels. router.PROVIDER also ignores AZDATA_LLM_PROVIDER (unlike nlsql.PROVIDER, which defaults to ollama), and /classify never passes a provider, so operators cannot redirect Task 2 to a local model. This is a single point of failure for all classification. *Fix:* Make the local tier a genuinely independent local provider (e.g. ollama) distinct from the strong cloud tier, both read from env (e.g. AZDATA_LOCAL_PROVIDER / AZDATA_STRONG_PROVIDER). Keep the existing strong->local fallback but ensure the local tier runs on independent infra so an OpenRouter outage still yields a (lower-quality) answer. Fail fast with a clear error when OPENROUTER_API_KEY is missing instead of constructing a client with api_key=None.

- **Failed classifications are cached as successes and tier errors are silently swallowed** `reliability` — `/Users/frodobaggins/Dev/AZdata/src/router.py` (classify_route lines 55-66 and classify_item lines 90-99; api.classify cache write at /Users/frodobaggins/Dev/AZdata/src/api.py lines 173-174) · When both tiers fail, classify_route returns chosen=local with label=None / confidence=0.0 and drops rag.classify_rag's 'error' field; classify_item's returned dict never contains an 'error' key under ANY circumstance. api.classify caches whenever `not result.get('error')` — which is always True even for total failures — so a label=None result produced during an OpenRouter outage is written to _classify_cache and then served (with cached:true) for every identical request thereafter, persisting the failure long after the provider recovers (until LRU eviction or process restart). The same masking applies to HS assignment: eqm.assign_code returns ok=False with a confidence-0.0 fallback code on LLM failure (eqm.py lines 163-167), but classify_item exposes only code/description, so a failed HS lookup is indistinguishable from a real one and is likewise cached as good. *Fix:* Propagate failure state: have classify_route/classify_item set an explicit ok/error field when the chosen tier has ok=False, and surface hs ok/confidence. In api.classify, only cache genuinely successful results (e.g. result.get('label') is not None and not result.get('error')). Consider a short negative-result TTL rather than permanent caching of failures.

- **Embedding HTTP call has no timeout and no retry — can block a worker thread indefinitely** `reliability` — `/Users/frodobaggins/Dev/AZdata/src/rag.py` (embed_texts line 35 (urllib.request.urlopen(req) with no timeout); reached at request time via rag.retrieve line 76 and eqm.retrieve_codes at /Users/frodobaggins/Dev/AZdata/src/eqm.py line 60) · embed_texts calls urllib.request.urlopen(req) with no timeout= argument and no _with_retries wrapper, unlike nlsql.call_llm which passes timeout=LLM_TIMEOUT. Python's default socket timeout is None (verified: no socket.setdefaulttimeout anywhere in rag.py/eqm.py), so if the embedding backend (Ollama — default localhost but remote-capable via OLLAMA_HOST, and prone to long cold model loads / GPU stalls) accepts the connection but then stalls, the call blocks forever. The FastAPI /classify and /query handlers are synchronous `def` endpoints (verified), so they run on the bounded threadpool; a handful of hung embedding calls exhaust the pool and the entire API (including /health) stops serving. Embedding runs on every classify request (retrieval is the shared first step for both tiers), so this hang is squarely on the hot path. *Fix:* Pass timeout=LLM_TIMEOUT (or a dedicated AZDATA_EMBED_TIMEOUT) to the urlopen call and wrap the request in nlsql._with_retries (or equivalent) so transient embedding failures retry and genuine stalls abort instead of hanging the thread.

- **Over-broad retries that also compound with the OpenAI SDK's own retries, with no overall deadline** `reliability` — `/Users/frodobaggins/Dev/AZdata/src/nlsql.py` (_with_retries lines 102-113 (bare `except Exception`); call_llm openai/openrouter branches lines 134-148) · _with_retries catches bare Exception (the comment claims 'transient' but it is not filtered) and retries 3x with backoff for ALL errors, including permanent ones such as 400 invalid-request, 401 bad API key, or model-not-found — delaying the surfaced error and multiplying load on the upstream. For the openai/openrouter providers this stacks on top of the OpenAI SDK's built-in retries (default max_retries=2), so a persistent 429/5xx can fan out to roughly 3x3 attempts, each up to LLM_TIMEOUT (default 120s) seconds, with no global deadline; on the synchronous FastAPI threadpool this pins a worker for minutes. There is no jitter and no total time/attempt budget. *Fix:* Restrict the except to known-transient exception types (socket/timeout, connection errors, HTTP 429/5xx) and re-raise others immediately. Disable the SDK's internal retries (max_retries=0) when wrapping its calls in _with_retries, or drop the manual wrapper for SDK providers. Add an overall deadline / attempt budget and backoff jitter.

- **Transient upstream and DB failures are returned as HTTP 400, defeating client retries** `reliability` — `/Users/frodobaggins/Dev/AZdata/src/api.py` (/query lines 128-129; /classify lines 178-182; errors originate in nlsql.answer try/except at /Users/frodobaggins/Dev/AZdata/src/nlsql.py lines 326-328) · nlsql.answer funnels every exception (LLM timeout, OpenRouter 5xx/connection error, psycopg2 DB connection failure) into result['error'], and /query returns all of them as HTTP 400. /classify likewise wraps any router exception and returns 400. HTTP 400 signals a permanent client error, so well-behaved clients, proxies, and retry middleware will NOT retry what are usually transient, recoverable upstream outages, and monitoring cannot distinguish bad input from provider/DB downtime. This is most damaging precisely when the single cloud dependency (see the OpenRouter finding) is flapping. *Fix:* Classify errors and map to appropriate status codes: 504 for LLM/DB timeouts, 502/503 for provider/connection errors, and reserve 400 for genuine bad input (guard rejections, malformed ref_date). Add an error category/type field to the response body for observability and client retry logic.

- **Prompt optimizer selects on, and reports, the same dev sample — no held-out validation, accepts noise-level gains** `ml_eval` — `/Users/frodobaggins/Dev/AZdata/scripts/optimize_prompt.py` (main() lines 152-174 (selection 'if cand_acc > best_acc' line 162; writes/reports best on line 170-173), evaluate() lines 94-123, load_dev_sample() lines 36-59) · The optimization loop evaluates the baseline and every candidate instruction on one fixed ~250-row dev sample, greedily keeps any candidate with cand_acc > best_acc, and then writes best_instructions.txt and prints 'best: X%' — all on the SAME sample used for selection. This is in-sample selection: the reported best accuracy is optimistically biased and the chosen prompt is overfit to those 250 rows (with k=16 retrieval the sample is also ~20% leaked per finding 1). Acceptance has no margin or significance test, so a +1-example change (0.4% on 250 rows) is treated as a real improvement over 4 rounds, fitting sample noise. Separately, load_dev_sample uses equal-per-stratum allocation ('per = sample_size // len(strata)', line 53), so the optimized metric is balanced accuracy, not the deployment (proportional) distribution. *Fix:* Hold out a separate validation split (disjoint from the proposer's error-feedback set) and select the winning instruction on it; report the final chosen instruction's accuracy on the untouched test set via eval_task2.py, never on the optimization sample. Require a minimum margin / paired-bootstrap significance before accepting a candidate. If the target metric is overall accuracy, sample proportionally rather than equally per stratum.

- **EQM HS-code assignment has no ground truth and no evaluation — accuracy is never measured** `ml_eval` — `/Users/frodobaggins/Dev/AZdata/src/eqm.py` (_selftest() lines 177-183 (prints only, no comparison) and assign_code() lines 125-167) · assign_code returns an HS code plus a confidence for every Good item, but there is no labeled HS-code gold set and no evaluation path anywhere. _selftest merely runs 6 test rows through assign_code and prints code/description/confidence with nothing to compare against. There is no top-1 accuracy, no candidate recall@k, and no separation of retrieval recall (is the correct code even in the candidate pool) from selection accuracy. The entire HS-code deliverable is therefore unverified — confidence values are emitted but their correctness is unknown. *Fix:* Create an adjudicated gold set of (item_text -> correct HS code) for a few hundred items and add a metric harness that reports top-1 code accuracy and candidate recall@k, reporting retrieval recall and LLM-selection accuracy separately. At minimum, log how often the LLM-chosen code is valid vs falls back to cands[0], since that fallback is currently invisible.

- **EQM heading pre-filter can permanently exclude the correct code, and returned confidence can belong to a rejected code** `ml_eval` — `/Users/frodobaggins/Dev/AZdata/src/eqm.py` (assign_code() lines 138-142 (heading filter / fallback) and lines 157-161 (substitution + confidence)) · Two issues in assign_code undermine HS-code correctness and confidence validity. (1) When predict_headings returns any headings, candidates are restricted to codes under those 4-digit headings (line 140), and the embedding fallback runs only 'if not cands' (line 141) i.e. only when the filtered set is EMPTY. So if the LLM predicts a non-empty but wrong heading set, the true code's heading is never in the candidate pool and can never be selected — a single LLM heading mistake is unrecoverable, with no union of heading-filtered and embedding candidates. (2) When the LLM returns a code not in the candidate list, the code is silently replaced by cands[0] (line 158) but the returned confidence is still float(parsed.get('confidence')) (line 161) — the model's confidence about its own rejected code, not about the cands[0] code actually returned. Confidence is thus decoupled from the output, and there is no ground truth (finding 3) to catch either problem. *Fix:* Always union the heading-filtered candidates with the top-k embedding candidates so a wrong heading cannot drop the true code from the pool. When substituting cands[0] for an invalid LLM code, reset confidence (e.g. to 0 or recompute) and/or set ok=False so the reported confidence corresponds to the code actually returned.

- **Non-numeric confidence string discards the model's chosen HS code** `correctness` — `/Users/frodobaggins/Dev/AZdata/src/eqm.py` (assign_code, line 161 (success-path return inside the try at lines 137-167)) · After the LLM successfully picks a valid candidate code, the result dict builds confidence with `float(parsed.get("confidence", 0) or 0)`. If the model emits a non-numeric confidence (e.g. "high" or "0.9 (likely)"), `float()` raises ValueError, which is swallowed by the broad `except Exception` at line 163. The handler then returns `cands[0]` with ok=False and confidence 0.0 — i.e. it throws away the code the model actually chose and substitutes the top embedding/heading candidate (often a DIFFERENT code). Reproduced: parsed={'code':'2921430000','confidence':'high'} -> ValueError -> fallback. router.classify_item surfaces hs.get('code') regardless of ok, so the wrong code is silently reported. The codebase already has a robust clamp (classify.py `_normalize_confidence`) that is not reused here. *Fix:* Parse confidence defensively before/independently of choosing the code, e.g. reuse `classify._normalize_confidence(parsed.get("confidence"))`, or wrap just the float() in its own try/except defaulting to 0.0. Do not let confidence coercion abort the whole (already-successful) classification.

### Low

- **Table allowlist ignores the schema qualifier (only the bare table name is checked)** `security` — `/Users/frodobaggins/Dev/AZdata/src/nlsql.py` (guard_sql table loop, lines 232-239 (name = table.name.casefold())) · The allowlist compares exp.Table.name, which is only the final identifier; table.db (schema) and table.catalog are ignored. So 'SELECT * FROM anyschema.einvoice' passes because name='einvoice' is allowed, regardless of schema. expression.sql() preserves the schema qualifier, so the executed query targets the attacker-chosen schema. If any same-named table exists in another schema the DB role can read (staging/backup/other-tenant/restore schemas), it becomes reachable, defeating the intended single-schema restriction. This compounds with the function-call gap above. *Fix:* Validate the fully-qualified name: reject any exp.Table that carries a non-empty db/catalog unless it matches an explicitly allowed schema, and compare against schema.table keys from the catalog. Alternatively, pin a fixed search_path on the connection and require unqualified table references only (reject any Table node that has a schema part).

- **Internal exception text returned to clients (information disclosure)** `security` — `/Users/frodobaggins/Dev/AZdata/src/api.py` (/query returns full result incl. error at line 129 (error built in nlsql.py answer, line 327); /classify returns {'error': str(exc)} at lines 178-182) · Raw exception messages are serialized into HTTP response bodies. nlsql.answer sets result['error'] = f"{exc.__class__.__name__}: {exc}" (nlsql.py line 327), and /query returns that whole dict with status 400 (api.py line 129); /classify returns {'error': str(exc)} (api.py line 182). psycopg2/DB errors, sqlglot parse errors, and filesystem paths thus reach the client, leaking schema details (table/column names, types) and internal paths that help an attacker map the database and probe the guard. The /query response also echoes raw_sql, the unguarded model output. *Fix:* Return a generic client-facing error plus a server-side correlation id, and log full exception detail server-side only. Avoid echoing raw_sql in non-debug responses, and gate verbose error output behind an explicit debug flag.

- **Column allowlist is not table-scoped and is bypassable (SELECT * and alias-qualified columns)** `security` — `/Users/frodobaggins/Dev/AZdata/src/nlsql.py` (guard_sql column loop lines 241-250; _catalog_columns lines 200-201) · _catalog_columns builds a flat union of all column names across all tables (not bound to a table). The column loop then skips the star projection (line 244, 'if column.name == "*": continue') and accepts any column whose qualifier matches a table alias: for 'SELECT e.<anything> FROM einvoice e', name is not in known_columns but column.table ('e') is in aliases, so neither GuardError at lines 247-250 fires. Because the catalog is parsed from the full DDL (db/schema.sql), the practical confidentiality exposure is limited to columns present in the live DB but absent from schema.sql (schema drift) and PostgreSQL system columns (ctid, xmin, xmax, cmin, cmax, tableoid). Impact is low on its own, but the column allowlist therefore provides essentially no confidentiality boundary and gives a false sense of restriction. *Fix:* Resolve each column's alias/qualifier back to its concrete table and verify the column belongs to that table's catalog columns (table-scoped check rather than a flat name set). Reject bare '*' and 'table.*' by requiring explicit catalog columns, and reject any qualified column whose resolved (table, column) pair is not in the catalog.

- **Escalation gated solely on miscalibrated self-reported LLM confidence** `reliability` — `/Users/frodobaggins/Dev/AZdata/src/router.py` (classify_route line 43 (THRESHOLD=0.9 at line 14); confidence source in rag.DEFAULT_INSTRUCTIONS at /Users/frodobaggins/Dev/AZdata/src/rag.py lines 88-90) · The only trigger for escalating to the strong/cloud model is `local.confidence < 0.9`, where confidence is a number the local LLM emits about itself (the prompt literally says 'confidence = your probability the label+group are correct'). Self-reported LLM confidences are poorly calibrated and tend to cluster at round high values (0.9/0.95/1.0), so a confidently-wrong local model reports >=0.9 and never escalates (silent misroute), while an honest model over-escalates and inflates cloud cost/latency. The strong tier's own confidence is never checked (classify_route only inspects ok at line 55), so escalation gives no quality guarantee. There is no neighbor-agreement, consistency, or calibration check even though retrieve() already computes the cosine similarities that could provide a calibrated signal. *Fix:* Stop using raw self-reported confidence as the sole gate. Incorporate retrieval-based signals already available in retrieve() (top-k neighbor label agreement / similarity margin), escalate when the local label disagrees with the retrieved-neighbor majority, treat a missing confidence field as low confidence, and calibrate the threshold against the dev set rather than hardcoding 0.9.

- **Confidence is collected but never calibrated; error paths hard-set confidence to 0.0** `ml_eval` — `/Users/frodobaggins/Dev/AZdata/scripts/eval_task2.py` (evaluate_row() line 67 and print_report() lines 113-160 (confidence written/printed, no calibration); confidence is self-reported per src/rag.py DEFAULT_INSTRUCTIONS line 89, and forced to 0.0 on failure in rag.py line 137 / eqm.py line 166) · The model is prompted to emit 'confidence = your probability the label+group are correct' (rag.py line 89), and eval_task2 records it (line 67), writes it to eval_results.csv, and prints it next to misclassified examples (line 159). But no calibration is ever computed: there is no ECE/Brier/reliability-bin or AUROC of confidence vs label_correct. Self-reported LLM probabilities are surfaced as if trustworthy. Worse, every exception/parse-failure path hard-sets confidence to exactly 0.0 (rag.py line 137; eqm.py line 166), so those rows would corrupt any reliability curve if one were later computed. *Fix:* Add calibration metrics to print_report (reliability bins + ECE + Brier, and AUROC of confidence vs correctness), excluding or separately bucketing the synthetic 0.0 error confidences. If confidence is to be used downstream (e.g. routing/escalation), fit a post-hoc calibrator (temperature/Platt scaling) on a held-out split rather than trusting raw self-reported values.

- **Confusion matrix counts parse failures and unknown labels as 'Service'** `ml_eval` — `/Users/frodobaggins/Dev/AZdata/scripts/eval_task2.py` (print_confusion() lines 100-110 (mapping at lines 104-105)) · print_confusion collapses any true_label or pred_label not exactly in ['Good','Service'] to 'Service'. ## Themes

- Silent, fail-wrong-quietly behavior is the through-line. Broad `except Exception` handlers and fallbacks discard the model's real answer (HS confidence coercion, out-of-list code substitution), null valid outputs (group casing/whitespace, the active-row filter's brittle string match), cache outright failures as successes, trust miscalibrated self-reported confidence, and tally parse failures as 'Service' — so the system degrades to a plausible-but-wrong result instead of erroring loudly.
- Security boundaries give a false sense of safety. The SQL guard is bypassable on function calls, schema qualifiers, and column scoping, and is treated as the sole control with no least-privilege DB role; the API adds no auth, no rate limiting, wildcard CORS, and leaks internal exception text and raw model SQL. Defense-in-depth is absent where it matters most.
- The evaluation cannot be trusted. Verbatim train/eval data leakage, in-sample prompt-optimizer selection with no held-out validation and noise-level acceptance, no HS-code ground truth at all, no confidence calibration, and a confusion matrix that conflates parse failures with Service mean the reported numbers do not measure generalization.
- External-dependency handling is fragile and inconsistent. The embedding hot path has no timeout or retry and can hang a worker forever, while other calls retry every exception (including permanent 4xx) and stack with the SDK's own retries with no overall deadline; transient outages are mapped to HTTP 400 that defeats client retries; and a single hardcoded cloud provider is a system-wide single point of failure.
- Pervasive per-request recompute and no resource reuse. A fresh DB connection (plus a redundant second one) per request, LLM clients rebuilt on every call, float32 indexes upcast to float64 and fully sorted on every query, retrieval repeated on every tier escalation, and the static schema block re-rendered each request — individually low severity, collectively a consistent efficiency-hygiene gap.
- Duplication, drift, and thin tests on the critical paths. A dead divergent copy of the router, a reimplemented JSON parser, duplicated model-resolution logic, dead constants/fields, an undeclared 'anthropic' dependency, and no unit coverage on the security-critical SQL parser/guard branches or the router escalation logic. Notably, the correct, tested helper almost always already exists in the repo — it simply is not reused, which is also why the fixes are mostly small.
